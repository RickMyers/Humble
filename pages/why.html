In this day and age, using a framework, and really any framework, comes down to a few simple question:

Why would I want to use this thing, and what does it offer me?

Here are a few reasons why you might want to take a look a bit further:

  • You understand Separation of Concerns
    • With Humble, Controllers are written in XML, Models in PHP, and Views in any of a list of templating language. Also, those REST/SOAP calls are implemented using yaml, so complicated coding isn't required. This establishes clear demarcation between each tier of the application, and allows delineation of responsibilities based upon the underlying technology.
  • You understand convention over configuration
    • With Humble, conventions are strictly enforced, so you only need to provide the parts of the application that make your application unique. Follow conventions and "magic" things happen.
  • You realize the best code is no code
    • Humble uses abstraction, code generators, and conventions to make common tasks easy, even "magical". This means that the developer does not have to provide code for every little thing. The less code you have to write to implement something, the less time will be taken supporting it, and fewer bugs will find their way into your application. Humble also documents, catalogs, and categorizes your code, which assists with reusability of components.
